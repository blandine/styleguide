1Style Rules

Here are detailed all the style rules for the web application.

1.1Depth

Getting a deeper look into the styles of shadows and positioning. The first thing we need to understand is the concept of stacked material (see http://goo.gl/A2ZAOH ). Keep the number of stacked layers to a minimum and the total number of layer below 3 (including the background). If a raised element lives inside of another raised element it should have a higher material-depth number (i.e a panel with a depth of 1 can have one panel with a depth of 2 above (inside) it.) Elements floating above the main layout (such as context menus) should have the highest material-depth number possible (2).

Example
Background
Layer 1
Layer 2

Stacked 

<div class="spacer"> Background </div>
<div class="spacer material-depth-1"> Layer 1 </div>
<div class="spacer material-depth-2"> Layer 2 </div>
<div class="spacer material-depth-1">
<p class="material-depth-2"> Stacked </p>
</div>

1.2.1Trench

The trench works much like the depth, however since it gives the illusion of a layer behind the current pice of material it should not be used on large areas, this will create raised "frames" around the trench.

Example
Layer 1

1.2Lists

A list can have several options attached to it in a so called 'leave-behind' (http://goo.gl/DMxAHS), however, options can only effect the list item they are a part of. Individual options can have a tooltip and an icon and they are displayed only if the item is hovered or clicked. If the list is long enough to extend outside of the main workarea it should be scrollable to avoid making the main view scroll.

If there are actions that affect the whole list (such as create new or delete all) they should be placed above or below the list.

Example
  • List Item 1

    Options

  • List Item 2
  • List Item 3

    Right behind

1.4.8Sidenav toggles

Sidenav toggles are a pair of checkboxes that controls the visibility of the sidenavs. The toggles are not part of the sidenav (if they were they would be hidden aswell and the user would be unable to show the sidenav without reloading the page). They instead trigger a Angular ng-hide on their respective sidenav.

Example 
<div class="sidenav-toggle">
<div class="sidenav-wrapper left">
<label for="nav-right">
	<i class="mdi mdi-fast-forward"></i>
</label>
</div>
</div>
<div class="sidenav-toggle">
<div class="sidenav-wrapper right">
	<i class="mdi mdi-rewind"></i>
</div>
</div>

1.6Buttons

When using buttons, avoid non specific actions as a label. For example, "OK" and "GO" are not specific while "SAVE USER" and "GO ON DASHBOARD" are. This helps the user understand what the action will do and what object it will effect.

1.6.1Flat buttons

The flat buttons are 'printed' on the material and only show its area when hovered, it uses capital letters to differentiate it from a link.

Example 
<button class="wfm-btn wfm-btn-invis-default">DEFAULT</button>
<button class="wfm-btn wfm-btn-invis-primary">PRIMARY</button>

1.6.2Raised buttons

The raised buttons area is always visible.

Example 
<button class="wfm-btn wfm-btn-default material-depth-1">Default</button>
<button class="wfm-btn wfm-btn-primary material-depth-1">Primary</button>
<button class="wfm-btn wfm-btn-invis-disabled">DISABLED</button>
<button class="wfm-btn wfm-fab">+</button>

1.8Tables

Tables are used to give a overview of data, however, following the Main concepts of the styleguide, only relevant data should be displayed. In the cases where that is not possible, please use infinite scrolling or pagination to keep the page lightweight. If a table do take time to be loaded, a loading bar should be displayed during this time to indicate to the user that it will take some time and that the table is not broken. A table should be displayed in alphabetical order by default, unless there is an obvious way the user would want the data sorted. Even so, the ability to sort and search within the table is most welcome.

1.10Notices

Notices indicate to the user that an event or action has taken place or that something is stopping the current action. There are several different markers to make the type of notice easy to spot:

  • The default one is used to provide general information to the user.
  • Details are never shown alone, but rather beneath one of the other markers.
  • The warning notice should be used to alert the user that something have to be done before they can continue.
  • The error notice should only be triggered when something goes wrong in the background.

A notice should either self-destruct or be dismissable to free up space.

Example
Hello, let me tell you about notice blocks.
I was born to show details.
Make sure to read all of the styleguide.
Something triggered this error notice.
<div class="notice-group">
<div class="notice default">Hello, let me tell you about notice blocks.</div>
<div class="notice detail">I was born to show details.</div>
</div>
<div class="notice warning">Make sure to read all of the styleguide.</div>
<div class="notice error">Something triggered this error notice.</div>

1.14Menus

1.14.1Context menus

Context menus should be used carefully and only where relevant. In the cases where they are used, limit the number of actions displayed within to a maximum of 5. If more options are needed, use a 'select input' instead. Please note that a menu item can not open another menu and there are no nested menus.

Example
Sign in Add a new skill Drop agents table
Sign out Add a new agent Drop agents table 
<div class="dropdown" tabindex="0">
<i class="mdi mdi-menu"></i>
<div class="inner material-depth-2">
	<i class="mdi mdi-account"> Sign in</i>
	<i class="mdi mdi-plus"> Add a new skill</i>
	<i class="mdi mdi-alert"> Drop agents table</i>
</div>
</div>
<div class="dropdown" tabindex="0">
<i class="mdi mdi-menu"></i>
<div class="inner right material-depth-2">
	<i class="mdi mdi-account"> Sign out</i>
	<i class="mdi mdi-plus"> Add a new agent</i>
	<i class="mdi mdi-alert"> Drop agents table</i>
</div>
</div>
